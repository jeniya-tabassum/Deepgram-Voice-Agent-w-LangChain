Why this demo exists

Deepgram does the voice.
The agent does the reasoning.

  • Deepgram Voice Agent — speech-to-text, text-to-speech, turn-taking, barge-in. Hard real-time audio, solved.
  • LangChain agent — breaks a complex question into a sequence of tool calls (RAG, a calculator, inventory, custom functions) and chains them into one answer.
  • The point: wiring them is one function + one WebSocket message — and the agent can reason through as many steps as it needs.

The architecture

Browser mic ──PCM16──▶ FastAPI ──WS──▶ Deepgram Voice Agent (STT · router LLM · TTS) Browser 🔊 ◀──PCM16── FastAPI ◀──WS── │ │ needs a real answer? ▼ FunctionCallRequest: ask_support_brain │ ▼ 🧠 LangChain agent (agent⇄tools loop) reasons step-by-step, chaining tools: search · inventory · calc_payment · trade_in · book (agent picks order) │ ▼ FunctionCallResponse ──▶ Deepgram speaks it

The one idea to take away

Two LLMs. Two jobs.

Inside Deepgram
Gemini — the Router

Decides when to reach for the brain. Speaks the result aloud. Never invents policy.

Inside LangChain
OpenAI — the Agent

Decides which tools to call and in what order, chains them across multiple steps, grounds the answer, returns short speakable text.

Swap the tools, the retriever, or the model — without touching one line of the voice pipeline.

The entire integration

One function. No endpoint. No tunnel.

  • Deepgram is configured with exactly one tool: ask_support_brain  deepgram_agent.py:152
  • It has no endpoint → it's a client-side function. Deepgram calls back over the same WebSocket.  :178
  • The prompt tells Gemini: "always ground answers in ask_support_brain."  :94
  • Result: no public URL, no ngrok — the whole thing runs on localhost.

The handoff, in three lines

1 · Deepgram asks

FunctionCallRequest arrives on the WS  — main.py:107

2 · Agent answers

run_support_brain(…, on_event=…) streams each step  — main.py:80

3 · We reply

FunctionCallResponse back on the same socket  — main.py:88

That round-trip is the entire integration.

No new connection. No REST endpoint. The socket that carries audio also carries the brain call.

Inside the brain — a multi-step tool-calling agent

┌───────┐ the agent loops: think → call a tool │ START │ → read result → think → … → answer └───┬───┘ ▼ ┌─────────────────────────────┐ ┌───────┐ call │ search_info · RAG │ │ agent │ ─────▶ │ check_inventory · price │ │ (LLM) │ ◀───── │ calc_payment · financing calc │ └───┬───┘ result │ estimate_trade_in · fn │ ▼ │ book_appointment · workflow │ ┌───────┐ └─────────────────────────────┘ │ END │ └───────┘ e.g. "trade in my Civic, monthly on a RAV4?" → estimate_trade_in → check_inventory → calc_payment → answer

The agent chooses which tools and in what order per question — 1 step or 3. The UI lights each tool live with a step trace. Add tools, APIs, sub-agents — Deepgram never changes. langchain_brain.py → build_support_agent()

"Isn't an LLM + a function call enough?"

Function calling is the mechanism.
The agent is what uses it well.

  • The step count depends on the question — "how does financing work?" needs 1 tool; the trade-in question needs 3. The agent plans that per query; hard-coded routing can't cover the combinatorial space.
  • Chaining in one voice turn — inventory price feeds the calculator; the trade-in value becomes the down payment. Push those up to Deepgram and each is a separate voice round-trip.
  • Add tools, external APIs, sub-agents, tracing — the agent grows without ever changing the voice integration.
